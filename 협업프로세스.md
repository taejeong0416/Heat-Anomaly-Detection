# 멀티 에이전트 협업 프로세스

---

## 에이전트 구성

| 역할 | 전문 영역 | 주요 관점 |
|------|-----------|-----------|
| **통계 전문가** | 통계 모델링, 검정, 분포 분석, 이상치 탐지 방법론 | 통계적 타당성, 가정 검증, 방법론 적절성 |
| **난방/열 도메인 전문가** | 열 원격검침, 난방 시스템, 에너지 사용 패턴 | 도메인 현실성, 실무 적용 가능성, 이상 유형의 실제 의미 |
| **데이터분석 전문가** | 데이터 엔지니어링, ML 파이프라인, 시각화 | 구현 효율성, 데이터 품질, 코드 최적화 |
| **최종 팀장** | 프로젝트 관리, 의사결정 종합 | 일정, 우선순위, 최종 의사결정, 결과 취합 |

---

## 협업 플로우

각 Phase는 **계획 점검 → 분석 실행 → 결과 점검** 3단계로 진행한다.

```
┌─────────────────────────────────────────────────────┐
│                   Phase N 시작                       │
├─────────────────────────────────────────────────────┤
│                                                     │
│  ① 계획 점검 (Plan Review)                          │
│     각 전문가가 Phase 계획을 검토 → 토론 → 확정      │
│                                                     │
│  ② 분석 실행 (Execution)                            │
│     확정된 계획에 따라 분석 수행                      │
│                                                     │
│  ③ 결과 점검 (Result Review)                        │
│     각 전문가가 결과를 검토 → 토론 → 최종 승인       │
│                                                     │
│  ④ 팀장 취합 보고 (Report)                          │
│     팀장이 토론 결과 + 산출물 종합 정리               │
│                                                     │
└─────────────────────────────────────────────────────┘
```

---

## 단계별 상세

### ① 계획 점검 (Plan Review)

**목적**: Phase 시작 전, 분석 방향과 방법의 적절성을 다각도로 검증

**진행 방식**:
1. `분석플랜.md`의 해당 Phase 내용을 공유
2. 각 전문가가 자기 관점에서 리뷰 의견 제시
3. 토론을 통해 쟁점 해소
4. 팀장이 최종 계획 확정

**각 에이전트별 점검 포인트**:

| 에이전트 | 계획 점검 시 확인 사항 |
|----------|----------------------|
| 통계 전문가 | 통계적 방법론 적절성, 가정 충족 여부, 검정력, 대안 방법 제안 |
| 도메인 전문가 | 도메인 관점의 타당성, 현실 적합성, 놓친 도메인 규칙, 이상 유형 현실성 |
| 데이터분석 전문가 | 데이터 품질 이슈, 구현 가능성, 성능/메모리 제약, 파이프라인 설계 |
| 팀장 | 일정 적합성, Phase 간 의존성, 우선순위, 쟁점 정리 및 의사결정 |

### ② 분석 실행 (Execution)

- 확정된 계획에 따라 코드 작성 및 분석 수행
- 실행 중 예상치 못한 이슈 발견 시 즉시 공유 → 미니 토론 후 방향 수정

### ③ 결과 점검 (Result Review)

**목적**: 분석 결과의 품질, 해석의 정확성, 다음 Phase 영향도를 검증

**각 에이전트별 점검 포인트**:

| 에이전트 | 결과 점검 시 확인 사항 |
|----------|----------------------|
| 통계 전문가 | 결과의 통계적 유의성, 분포 가정 위반 여부, 오버피팅 위험, 신뢰구간 |
| 도메인 전문가 | 결과가 도메인 상식에 부합하는지, 오탐/미탐 패턴의 실무적 의미 |
| 데이터분석 전문가 | 코드 품질, 재현 가능성, 산출물 완성도, 다음 Phase 입력 데이터 적합성 |
| 팀장 | 핵심 발견사항 요약, 다음 Phase에 전달할 의사결정 사항, 리스크 |

### ④ 팀장 취합 보고 (Report)

팀장은 토론 결과를 아래 형식으로 정리한다:

```markdown
## Phase N 완료 보고

### 핵심 결과
- (3~5줄 요약)

### 토론 쟁점 및 결론
| 쟁점 | 통계 | 도메인 | 데이터 | 최종 결론 |
|------|------|--------|--------|-----------|

### 다음 Phase 영향 사항
- (Phase N+1에 반영할 의사결정)

### 리스크/유의사항
- (잠재 이슈)
```

---

## 토론 규칙

1. **근거 기반**: 의견 제시 시 데이터, 통계, 도메인 지식 등 근거를 함께 제시
2. **관점 명시**: 발언 시 어떤 관점(통계/도메인/구현)에서의 의견인지 명확히
3. **쟁점 시 팀장 결정**: 합의 불가 시 팀장이 각 의견의 trade-off를 정리하고 최종 결정
4. **기록 필수**: 토론의 핵심 쟁점과 결론은 노트북 마지막 셀 또는 플랜에 기록

## 적용 범위

| Phase | 계획 점검 | 결과 점검 | 비고 |
|-------|:---------:|:---------:|------|
| Phase 0 | - | - | 이미 완료 |
| Phase 1 | - | - | 이미 완료 |
| Phase 2 | O | O | **다음 단계** |
| Phase 3 | O | O | |
| Phase 4 | O | O | 모델링 핵심 단계 — 심층 토론 |
| Phase 5 | O | O | 이상 탐지 핵심 — 심층 토론 |
| Phase 6 | O | O | 해석·검증 — 도메인 전문가 비중 높음 |
| Phase 7 | O | O | 보고서 — 전체 정합성 검토 |

## 실행 방법 (Claude Code 내)

각 Phase 시작 시 사용자가 아래와 같이 요청:

```
Phase N 계획 점검 시작해줘
```

→ 4개 에이전트가 순차적으로 의견을 제시하고 토론 후 팀장이 정리

```
Phase N 결과 점검 시작해줘
```

→ 분석 완료 후 동일하게 4개 에이전트가 결과를 리뷰하고 팀장이 취합 보고
