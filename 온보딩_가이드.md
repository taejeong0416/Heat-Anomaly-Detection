## 1. 프로젝트가 풀려는 문제

### 현재 난방공사의 이상 탐지 방식

기존에는 **"전년 동월 대비 ±30% 초과하면 이상"** 이라는 단순 임계치 하나로 이상사용을 판단했습니다.

이 방식의 문제:
- 30% 이내로 서서히 증가하는 배관 누수는 못 잡음
- 야간에만 비정상적으로 쓰는 패턴(밸브 미폐쇄)은 총량이 정상이면 놓침
- 주말에 무인 건물이 평일처럼 돌아가는 것도 월간 합산하면 안 보임

### 우리가 하려는 것

**시간별(24차원) 사용 패턴**을 분석해서, "이 건물의 오늘 사용 패턴은 정상 범위를 벗어났다"를 자동으로 잡아내는 알고리즘을 만드는 것입니다.

핵심 아이디어: **정상 패턴을 먼저 정의하고, 거기서 벗어나면 이상으로 판단한다.**

### 왜 이게 어려운가

- **라벨이 없습니다.** "이건 이상이다"라고 태깅된 데이터가 없습니다. → 비지도 학습이 필수.
- **정상의 정의가 맥락에 따라 다릅니다.** 주택용 아파트의 정상과 업무용 빌딩의 정상이 완전히 다릅니다. 겨울과 여름도 다릅니다.
- **이상의 유형이 다양합니다.** 급증, 야간 이상, 패턴 이탈, 계절 역행 등 원인과 형태가 각각 다릅니다.

## 2. 데이터를 먼저 이해해야 하는 이유

이 프로젝트에서 가장 중요한 건 모델이 아니라 **데이터의 물리적 의미를 이해하는 것**입니다.

### 데이터가 만들어지는 과정

```
열병합발전소 (115~120°C 고온수 생산)
  ↓ 1차측 배관
건물 기계실 (열교환기로 1차/2차 분리)
  ↓ 2차측 배관
각 세대 온돌 (바닥 복사난방)
  ↓
열량계가 측정: 유량 × (공급온도 - 환수온도) = 열량(Gcal)
  ↓
원격검침(AMI)으로 시간당 적산값 전송 → 중앙 시스템 저장
```

**이걸 알아야 하는 이유**: 데이터의 이상값이 왜 생기는지 이해할 수 있습니다.
- 음수값 → 열량계 리셋, 역류, 통신 오류 (물리적으로 열을 마이너스로 쓸 수 없음)
- NaN → AMI 통신 두절, 배터리 소진, 열량계 고장
- 극단값 → 센서 오류에 의한 과다계량

### 데이터 구조

각 행 = **하나의 설비(열량계)가 하루 동안 시간별로 쓴 열량**

```
날짜 | 지사 | 설치(설비ID) | 종별 | 1시 | 2시 | ... | 24시 | 총사용량
```

- 5년간(2021~2025) 약 1,690만 행
- 설비 약 10,000대, 19개 지사
- 종별: 주택용(아파트 세대), 업무용(오피스빌딩), 공공용(관공서), 냉수용(지역냉방)

### 종별이 왜 중요한가

종별마다 사용 패턴이 **완전히 다릅니다**:

| 종별 | 피크 시간 | 주말 특성 | 왜 다른가 |
|------|-----------|-----------|-----------|
| 주택용 | 17~22시 (귀가 후) | 평일과 비슷 | 온돌 연속난방, 개인 생활 패턴 |
| 업무용 | 09~18시 (업무시간) | 사용량 급감 | HVAC 스케줄 기반 운영 |
| 공공용 | 업무용과 유사 | 감소 (휴관) | 공공시설 운영시간 |

→ 이 차이 때문에 **종별을 섞어서 분석하면 안 됩니다.** 업무용의 "정상"이 주택용에선 "이상"일 수 있습니다.

## 3. Phase 0: 통합 — 왜 먼저 합쳤는가

### 문제 상황

데이터가 60개 월별 CSV로 쪼개져 있습니다. 어떤 설비의 5년간 패턴을 보려면 60개 파일을 다 열어야 합니다.

### 한 일

- 60개 파일 → 단일 parquet로 통합 (16,868,714행)
- 총사용량을 1~24시 합으로 재계산 (원본 `총` 컬럼에 오차 가능성)
- 파생 컬럼 추가: 연도, 월, 일, 요일, 계절, 난방시즌, 공휴일
- `유형` 컬럼은 전 기간 `-`만 있어서 drop

### 중요한 판단: 이상값을 여기서 안 건드렸음

음수값 7,881건, 극단값 3,482건, NaN 104,369건이 있었지만 **이 단계에서는 제거하지 않았습니다.**

**이유**: 뭘 지울지 판단하려면 먼저 데이터를 봐야 합니다. "음수는 다 오류인가?" — 모릅니다. 종별마다, 크기마다 원인이 다를 수 있습니다. 성급하게 지우면 의미 있는 데이터를 잃을 수 있습니다.

→ 그래서 EDA(Phase 1)를 먼저 하고 전처리 기준을 정하기로 했습니다.

**산출물**: `data/processed/all_data.parquet`, `notebooks/01_preprocessing.ipynb`

## 4. Phase 1: EDA — 무엇을 봤고, 어떤 결론을 내렸는가

EDA의 목적은 "데이터를 이해하는 것"이 아니라 **"이후 단계의 의사결정을 위한 근거를 확보하는 것"** 입니다.

### 4-1. 종별 분포를 봤다 → 종별 분리 분석 결정

| 종별 | 비율 | 중앙값(Gcal) | 99.9%ile | 제로 비율 |
|------|:----:|:----------:|:--------:|:---------:|
| 주택용 | 49.8% | 3.09 | 63.4 | 1.7% |
| 업무용 | 38.3% | 0.18 | 51.2 | 37.4% |
| 공공용 | 10.6% | 0.13 | 81.8 | 34.8% |
| 냉수용 | 1.3% | 0.00 | 73.7 | 52.8% |

**이걸 본 의미**: 주택용 중앙값이 3.09인데 업무용은 0.18입니다. 17배 차이. 이걸 한꺼번에 정규화하면 업무용의 미세한 변동이 다 묻힙니다.

**결론**: 종별 분리 분석 필수. 냉수용은 1.3%밖에 안 되고 절반 이상이 제로이며 난방과 완전히 다른 시스템(지역냉방) → 분석 대상에서 제외.

### 4-2. 시간대 패턴을 봤다 → 이상 유형 정의의 기반

시간대별 평균 사용량 곡선을 그려보니:
- 주택용: 17~22시 피크 (귀가 후 난방), 09~17시 골 (외출)
- 업무용: 09~18시 피크 (HVAC 가동), 야간 거의 0
- 난방시즌 vs 비난방시즌 차이가 극명

**이걸 본 의미**: "정상 패턴"이 뭔지 눈으로 확인했습니다. 이걸 기준으로 "야간에 높으면 이상(야간 이상형)", "피크 시간이 아닌데 높으면 이상(패턴 이탈형)" 같은 유형을 정의할 수 있게 되었습니다.

### 4-3. 음수값을 분석했다 → 전처리 방침 결정

- 총 6,604건(0.04%), min = -73,039,997 Gcal
- min값이 -7,300만 → 이건 열량계 리셋(적산값 초기화)으로 밖에 설명이 안 됩니다
- 10시, 14시에 집중 → 검침(리셋/교체) 시간대와 관련

**이걸 본 의미**: 음수의 원인을 크기별로 구분할 수 있게 되었습니다.
- 극단적 음수(< -10,000 Gcal) → 검침기 리셋, 무조건 제거
- 중간 음수(총사용량 < 0) → 하루 전체가 오염, 제거
- 미세 음수(일부 시간만) → 역류 또는 보정, 0으로 clip

### 4-4. NaN을 분석했다 → 보간/제거 기준 결정

- 209,894건(1.2%)이 NaN 포함
- NaN=24(하루 전체 결측): 92,240건 → 보간 근거 없음, 제거
- NaN 1~2: 87,492건 → 1~2시간 통신 끊김(small gap), 전후 값으로 선형보간 가능
- NaN 3~23: ~28,000건 → 연속 3시간+ gap은 패턴 전환 구간(기상 상승 등) 전체를 소실시킬 위험, 제거
- 설비 중 45개는 30%+ 결측 → 열량계 장기 고장, 분석에서 제외

**이걸 본 의미**: "NaN이 몇 개까지면 살릴 수 있는가?" 에 대한 기준을 데이터에서 도출했습니다. EDA에서 연속 NaN 패턴을 분석한 결과 NaN 2개인 행의 99.9%, NaN 3개인 행의 89%가 연속 패턴이었고, 시계열 결측치 관련 문헌(Pelletier, 2024)에서도 선형보간은 small gap(1~2개)까지만 권장합니다.

### 4-5. 지사/기온 변수를 검토했다 → 기온 미사용 결정

- 지사가 익명화(Branch_0001~0019)되어 있어서 어떤 지역인지 모릅니다
- 지역을 모르면 해당 지역의 기온(HDD)을 매칭할 수 없습니다
- 대안으로 월/계절 변수가 기온의 proxy 역할 가능 (한국은 전국적으로 겨울에 춥고 여름에 더움)
- 지사 간 사용량 스케일 차이가 47배 → 지사 자체는 컨텍스트 변수로 유용

**결론**: 기온 직접 사용 불가, 월/계절로 대체. 지사는 컨텍스트 변수로 활용.

### EDA 전체 요약: 이 Phase에서 확정된 것

이 Phase의 가장 큰 성과는 **Phase 2 전처리의 모든 파라미터를 데이터 기반으로 확정한 것**입니다. 감으로 정한 게 아니라 분포를 보고 정했습니다.

## 5. Phase 2: 전처리 — 왜 이렇게 자르고 남겼는가

### 전처리의 원칙

**"이상사용 탐지 모델에 넣을 깨끗한 데이터를 만들되, 이상 패턴 자체는 보존한다."**

이게 핵심입니다. 전처리에서 제거하는 건 **장비 오류에 의한 쓰레기 데이터**이지, **사용 패턴이 이상한 데이터**가 아닙니다.

예시:
- 열량계 리셋으로 -7,300만 Gcal → 제거 (장비 오류)
- 야간에 사용량이 비정상적으로 높은 건물 → 보존 (이상사용 후보)

### 처리 순서와 각 단계의 이유

```
원본 16,868,714행
  │
  ├─ 냉수용 제외 (-211,988행)
  │   이유: 지역냉방 냉수는 난방과 완전히 다른 시스템. 포함하면 노이즈.
  │
  ├─ NaN 30%+ 설비 제외 (-53,258행, 45개 설비)
  │   이유: 5년 중 30% 이상이 비어있으면 열량계가 대부분 고장 상태.
  │         보간해봤자 원래 패턴과 무관한 가짜 데이터가 됨.
  │
  ├─ 검침기 리셋 행 제거 (-1,030행)
  │   이유: 시간당 < -10,000 Gcal은 적산값 초기화. 물리적 의미 없음.
  │
  ├─ 총사용량 음수 행 제거 (-1,411행)
  │   이유: 하루 전체 합이 음수 = 하루 전체가 오염된 데이터.
  │
  ├─ 시간대 음수 → 0 clip (4,667셀)
  │   이유: 총사용량은 양수인데 일부 시간만 음수 = 역류 또는 보정.
  │         하루 전체를 버리기엔 아까움. 해당 시간만 0으로.
  │
  ├─ 극단값 (threshold×100 이상) 행 제거 (-1,324행)
  │   이유: 시간당 259 Gcal(주택용 기준)은 정상의 100배. 검침 오류 확정.
  │
  ├─ 극단값 (threshold 초과) → clip (495,631셀)
  │   이유: 정상 범위의 tail. 완전 오류는 아니지만 모델 학습 왜곡 방지.
  │
  ├─ NaN=24 행 제거 (-60,341행)
  │   이유: 24시간 전부 결측 = 열량계 미작동 또는 통신 완전 단절.
  │         보간할 근거 0.
  │
  ├─ NaN 3~23 행 제거 (-28,253행)
  │   이유: 연속 3시간+ gap은 패턴 전환 구간(기상 상승 06→09시 등) 전체를
  │         소실시킬 수 있음. 선형보간은 small gap(1~2개)까지만 적절.
  │
  └─ NaN 1~2 → 선형 보간 (85,524행 보존)
      이유: 1~2시간 통신 끊김은 AMI에서 가장 흔한 결측 패턴.
            전후 값으로 복원 합리적. 보간 불확실성 낮음.
      
최종: 16,511,109행 (97.9% 보존)
```

### 극단값 threshold를 왜 "시간당 99.9%ile"로 잡았는가

- 총사용량이 아니라 **시간당 개별값** 기준입니다
- 왜: 총사용량으로 자르면 "시간당은 말도 안 되게 높은데 나머지 시간이 낮아서 총합은 정상"인 오류를 놓칩니다
- 왜 99.9%ile: 99%ile과 99.9%ile 사이는 정상 범위의 tail이고, 99.9%ile 이상부터 값이 급격히 뜁니다 (분포의 kink point)
- 결과: 주택용 2.59 Gcal/h, 업무용 2.74, 공공용 4.38

### 정규화: 왜 종별×난방시즌 6그룹인가

- 종별로 나누는 이유: 스케일이 다르니까
- 난방시즌으로 나누는 이유: 비난방시즌은 제로가 30~37%. 전체 기간 통합하면 mean이 낮아지고 std가 커져서 Z-score가 무의미해짐.
- 지사는 왜 안 넣나: 지사 내 설비 수가 적은 경우(수십 대) 소수 설비가 통계를 왜곡. 대신 Phase 4 Context Model에서 지사 효과를 흡수하는 설계.

### 데이터는 원본 스케일로 저장한 이유

정규화된 값으로 저장하면 나중에 "이 설비가 실제로 몇 Gcal 썼는지" 역추적이 불편합니다. 파라미터(mean, std)를 JSON으로 저장해두면 언제든 `(x - mean) / std`로 변환 가능. 유연성 확보.

|------|------|------|

| NaN 잔존 | 0건 | PASS |
| 제거율 | 종별별 3% 이내 | 주택 0.54%, 업무 1.16%, 공공 1.23% PASS |

## 6. 앞으로의 방향 (Phase 3~7)

### 전체 파이프라인 로직

```
[Phase 3] 패턴을 숫자로 표현 (Feature Engineering)
    ↓
[Phase 4] "정상"이 뭔지 학습 (정상 패턴 모델링)
    ↓
[Phase 5] 정상에서 벗어난 것을 찾음 (이상 탐지)
    ↓
[Phase 6] 찾은 이상을 유형별로 분류하고 설명 (해석)
    ↓
[Phase 7] 보고서
```

### Phase 3: Feature Engineering — 왜 필요한가

24시간 사용량(24차원 벡터)만으로도 패턴은 볼 수 있지만, **이상 유형별로 잡히는 신호가 다릅니다.**

| 이상 유형 | 잡으려면 이 Feature가 필요 |
|-----------|--------------------------|
| 급증형 | 전일 대비 변화율, 이동평균 대비 비율 |
| 야간 이상형 | 야간 비율 (22~06시 합 / 총) |
| 장기 미사용 후 급증형 | 연속 제로일 수 |
| 주말/공휴일 이상형 | 주말 대비 비율 |
| 기저 유량 이상형 | min(1~24시) — 최소 사용량 |
| 계절 역행형 | 비난방시즌인데 사용량 상위 |

또한 **음수(역류) 발생 빈도**도 설비별로 산출합니다. 이건 별도 이상유형이 아니라 보조 지표인데, 이유는:
- 음수는 `value < 0`으로 즉시 탐지 가능 (모델이 필요 없음)
- 하지만 역류가 잦은 설비에서 다른 이상(기저 유량 이상 등)이 함께 발견되면 → "배관/밸브 문제"로 해석하는 근거가 됨

### Phase 4: 정상 패턴 모델링 — 핵심 단계

**"이상 탐지는 정상의 정의가 핵심이다"**

두 가지 접근:
1. **클러스터링**: 종별별로 K-Means 등으로 "정상 패턴 프로토타입"을 만듦. "주택용은 보통 이런 모양이다"를 데이터에서 학습.
2. **Context Model**: 종별, 지사, 요일, 월 등을 입력 → 기대 사용량 예측. Residual(실제 - 기대) = 맥락 대비 어느 정도 벗어났는지.

### Phase 5: 이상 탐지 — 7개 유형

| # | 유형 | 한 줄 설명 | 실무적 의미 |
|:-:|------|-----------|------------|
| 1 | 급증형 | 어제보다 갑자기 엄청 많이 씀 | 배관 누수, 열량계 과다계량 |
| 2 | 야간 이상형 | 밤에 안 써야 하는데 씀 | 밸브 미폐쇄, HVAC 오류 |
| 3 | 패턴 이탈형 | 정상 곡선과 모양이 다름 | 설비 오작동, 용도 변경 |
| 4 | 장기 미사용 후 급증형 | 오래 0이다가 갑자기 급증 | 열량계 고장 후 재작동 |
| 5 | 계절 역행형 | 여름에 많이 씀 | 열손실, 배관 누수 |
| 6 | 주말/공휴일 이상형 | 업무용인데 주말에 평일처럼 씀 | 무인 건물 HVAC 미해제 |
| 7 | 기저 유량 이상형 | 아무도 안 쓰는 시간에도 일정량 흐름 | 배관 누수 |

탐지 방법:
- **Isolation Forest**: 전체 feature 공간에서 "혼자 동떨어진" 데이터 포인트 탐지
- **Autoencoder**: 정상 패턴으로 학습 → 복원 오차가 크면 이상
- 둘을 앙상블: 둘 다 이상이면 고신뢰, 하나만이면 저신뢰

### Phase 6: 해석 — SHAP으로 "왜 이상인지" 설명

이상으로 잡힌 샘플에 대해 "몇 시 사용량이 이상 판정에 가장 기여했는지"를 SHAP value로 시각화합니다.
비지도 학습이라 라벨이 없으므로, 수동 검토(유형별 10건)로 오탐을 확인하고 threshold를 재조정합니다.

## 7. 핵심 의사결정 요약 — "왜?"를 모아서

| # | 의사결정 | 왜 이렇게 했는가 |
|:-:|----------|-----------------|
| 1 | 종별 분리 분석 | 종별 간 스케일·패턴 차이가 17~47배. 섞으면 소규모 종별의 이상이 묻힘 |
| 2 | 냉수용 제외 | 1.3%, 제로 52.8%, 지역냉방은 완전히 다른 시스템. 포함하면 노이즈만 증가 |
| 3 | 기온 미사용, 월/계절 대체 | 지사 익명화로 지역별 기온 매칭 불가. 한국은 전국적으로 계절 변화 유사 |
| 4 | 음수값 전처리에서 제거 | 물리적으로 불가능한 값(장비 오류). 단, 발생 빈도는 Phase 3에서 Feature로 활용 |
| 5 | 음수(역류)는 별도 이상유형 아님 | `< 0` 규칙으로 즉시 탐지 가능. 모델 기반 탐지가 필요한 7개 유형과 레이어가 다름. Phase 6 해석에서 종합 보고 |
| 6 | "장기 미사용" 단독은 유형 불포함 | 급증 없이 계속 제로 = 설비 고장 또는 공실. "이상사용"보다 "미계량" 영역. 오탐 위험 높음 |
| 7 | NaN 1~2개만 보간, 3개+ 제거 | 선형보간은 small gap(1~2)까지 적절. 연속 3시간+은 패턴 전환 구간 소실 위험. 문헌 근거: Pelletier(2024) |
| 8 | 극단값 99.9%ile clip | 99.9%~99.99% 사이에서 값이 급격히 뜀(kink point). 99.9%ile이 정상 상한의 경계 |
| 9 | 정규화: 종별×난방시즌 6그룹 | 비난방시즌 제로 30~37% → 전체 기간 통합하면 Z-score 분포 왜곡 |
| 10 | 지사는 정규화 기준 미포함 | 소규모 지사에서 소수 설비가 통계 왜곡. Context Model에서 흡수하는 설계 |

## 알아두면 좋은 주의사항

1. **Branch_0007 (지사)**: NaN=24가 12,611건으로 압도적 1위. DCU(집중기) 장애로 추정. 이 지사에서 이상 탐지 결과가 나오면 통신 장애에 의한 오탐일 수 있음.
2. **비난방시즌 제로**: 업무용/공공용의 비난방시즌 제로 비율이 30~37%. 이게 "정상"인데, 모델이 이걸 이상으로 잡을 수 있음. 반드시 난방시즌 분리 모델링 필요.
3. **극단값 clip**: 약 49만 셀이 clip됨. 대형 건물의 정상 피크가 평탄화되었을 수 있음. 이상 판정 시 원본(`all_data_phase0.parquet`) 참고.
4. **라벨 없음**: 비지도 학습이므로 "정답"이 없음. 수동 검토 + SHAP 해석으로 신뢰도를 확보해야 함.
5. **데이터 비식별화**: 종별 단위로 랜덤 가중치 적용됨. 절대값은 원본과 다르지만 **상대적 패턴(비율)은 보존**. 종별 내 설비 간 비교는 유효.
